# redis-uya 方案评审

> 版本: v0.1.0-dev
> 日期: 2026-04-25
> 状态: 第二轮评审

## 1. 结论

`redis-uya` 的方向正确，但首版交付目标必须进一步收敛。项目的长期目标仍然是生产级、高性能、在核心场景上超过 Redis；首版目标则应是建立一个可编译、可测试、可交互、可恢复、可 benchmark 的单节点 Redis 内核。

当前最合理的路线是：

1. 把 `v0.1` 定义为单节点、RESP2、String/Key 命令子集、AOF 最小恢复闭环。
2. 把复杂类型、RDB 完整兼容、复制、集群、RESP3、淘汰策略放入后续版本。
3. 把“超越 Redis”作为持续 benchmark 驱动的版本目标，而不是首版唯一验收门槛。
4. 把“技术可行性”和“完整性”作为首版管理红线：任何能力上线前都必须先证明能实现、能测试、能收口。

## 2. 当前现实约束

### 2.1 编译器约束

当前项目内置的是 `Uya v0.9.4` 的本地修复快照。现阶段可以稳定支持：

- 工具模块和单元测试
- 内存分配器封装
- 配置解析
- SDS 基础实现
- 结构体切片字段相关 C99 代码生成

但仍不适合在首版大规模押注：

- 复杂泛型容器设计
- 大面积 async/await + epoll 联动
- `fork()` 驱动的后台持久化
- 需要编译器复杂推断的抽象接口层

结论：首版设计必须以“当前编译器能稳定承载”为前提。

### 2.2 产品约束

项目目标很大，但用户真正需要的是可运行、可测量的系统，而不是一份一次性覆盖全部 Redis 特性的设计图。首版若同时追求：

- 五大核心类型
- RESP2 + RESP3
- RDB + AOF + BGSAVE
- 主从复制
- 基础集群
- 首版性能超越 Redis

则范围明显失控。

## 3. 版本收敛建议

### 3.1 `v0.1` 必做

- 单进程、单节点
- RESP2 子集
- 命令：`PING`、`GET`、`SET`、`DEL`、`EXISTS`、`EXPIRE`、`TTL`、`INFO` 基础段
- 数据结构：String + 基础键空间
- 存储：项目内专用 `Dict`
- 引擎：单 DB 起步，可保留多 DB 接口
- 持久化：AOF append + replay
- 集成：`redis-cli` smoke + Python 子集测试
- benchmark：建立同机 Redis 对照基线

### 3.2 `v0.2` 建议做

- Hash/List/Set/ZSet
- `SCAN`
- 过期主动清理
- RDB 子集
- 更完整的 INFO / CONFIG
- 更完整的集成测试

### 3.3 `v0.3` 再做

- RDB 完整化
- `BGSAVE` / `BGREWRITEAOF`
- rewrite 路径完整化

### 3.4 `v0.4` 到长期目标

- `v0.4`: 主从复制
- `v0.5`: RESP3、事务、Pub/Sub、控制面增强
- `v0.6`: `maxmemory`、淘汰策略、Slab、内存治理
- `v0.7`: 基础集群
- `v0.8+`: 零拷贝、批量解析、SIMD、io_uring、核心场景超越 Redis

## 4. 设计修正点

### 4.1 `Dict` 不再以泛型 `HashMap<K, V>` 为前提

当前设计中的泛型 `Dict<K, V>` 过于超前。首版应先实现项目内专用字典：

- 键类型固定为 `Sds`
- 值类型固定为 `RedisObject` 或对象句柄
- 先实现 `insert/lookup/delete/len`
- 渐进式 rehash 作为第二阶段增强，而不是起步门槛

### 4.2 持久化先 AOF，后 RDB

RDB 完整兼容的二进制格式风险高，AOF 更适合早期恢复闭环。

首版建议：

- 命令写入 AOF
- 启动时回放 AOF
- 先不做 rewrite/fork

### 4.3 首版网络模型优先“能跑通”

事件循环应从稳定可调试出发：

- TCP listener
- 连接状态机
- 请求读取 / RESP 解析 / 命令执行 / 响应写回
- 100ms server cron

是否上更复杂 async 抽象，取决于实测收益，而不是先验设计。

### 4.4 性能目标分层

性能定义分三层：

- `floor`: 同机 Redis 基线的最低比例门槛
- `target`: 接近或追平 Redis
- `stretch`: 在特定核心场景超过 Redis

这样既不削弱项目目标，也避免首版目标失真。

## 5. 当前进度判断

目前已经完成并验证：

- 工程骨架
- `Makefile` / `build` / `test` 基础流程
- 日志、时间、字节序、CRC64
- 内存分配器封装
- 最小配置解析器
- SDS 基础能力
- 本地 Uya 编译器关键 bug 修复与同步

这说明项目已经具备进入 `v0.1-alpha` 主线开发的条件。

但进入主线后仍要坚持两条硬约束：

- 任何超出当前工具链承载能力的设计，不进入主实现。
- 任何缺少错误路径、释放路径、恢复路径和测试证据的能力，不算完成。

## 6. 立即行动项

1. 把 TODO 改成以 `v0.1-alpha / beta / release` 为主线。
2. 先做项目内专用 `Dict`，不再追求首版泛型化。
3. 尽快建立 `RESP2 + router + String/Key` 的最小协议闭环。
4. 在 `docs/` 内新增 DoD 与 benchmark 格式文档，约束后续验收。
5. TODO 必须覆盖全版本路线图，但当前执行只聚焦近端版本。
6. 所有后续模块继续坚持“先测试、再实现、再回归”。

## 7. 最终判断

这个项目仍然值得继续推进，但要按“先内核、再功能、后性能扩张”的顺序来做。当前最优策略不是把设计继续铺大，而是把 `v0.1` 收敛成一个稳定、可验证、可基准化的单节点 Redis 内核。
