# redis-uya 开发规范

> 版本: v0.1.0-dev
> 日期: 2026-04-25
> 适用阶段: Phase 0+

> 后续实现 `redis-uya` 时，默认遵循本文，除非任务文档明确覆盖。

## 1. 基本原则

- **生产级目标**：所有设计必须服务于 Redis 6.2+ 协议兼容、低延迟、高吞吐、可恢复、可观测、可维护，长期性能目标是在核心场景超越 Redis。
- **技术可行性优先**：任何设计、抽象和接口都必须先经过当前工具链、当前代码库和当前测试体系验证，不以愿景替代落地能力。
- **完整性强约束**：不接受“核心路径能跑，但错误路径、释放路径、恢复路径、测试证据缺失”的交付物；完整性与功能正确性同级。
- **收敛优先**：不要一次性堆叠高级能力。先做可验证的最小闭环，再基于 benchmark 和测试扩展。
- **Uya-ready**：详细设计里的概念接口必须落成当前 Uya v0.9.4 本地修复快照可编译形态，不假设尚未发布的 v0.9.5 能力。
- **无绝对路径**：源码、脚本、文档命令默认使用工程相对路径，例如 `./uya/bin/uya`、`make build`。
- **显式边界**：文档中区分生产目标、当前阶段能力、临时限制和后续增强，避免把原型数据当作生产 SLO。

## 2. Uya 编码规则

- **语法保守**：只使用当前内置 Uya 编译器已支持并可验证的语言能力；拿不准时先写最小可编译代码。
- **入口约定**：应用入口使用 `export fn main() i32` 或 `export fn main() !i32`。
- **类型显式**：`const`/`var` 声明写明类型；不用 `let`、`mut`、`i++`、通配导入或臆造的 `std.*` API。
- **错误显式**：可失败路径使用 `!T`、`try`、`catch`；不依赖隐式 panic；错误路径同步测试。
- **资源清晰**：区分 owning type、borrow type、guard type；拥有资源的结构体必须有明确释放/回滚路径。
- **defer/errdefer**：资源释放用 `defer`，错误回滚用 `errdefer`；块内不写 `return`、`break`、`continue`。
- **FFI 谨慎**：日常指针用 `&T`/`&const T`，FFI 边界才用 `*T`/`*const T`；C 布局、对齐和 union 按 `union_memory_layout.md` 验证。
- **字符串安全**：字符串字面量优先接收为 `&[const byte]`；不要把字面量赋给元素可变 `&[byte]`。
- **模块规则**：`use` 只放文件顶层；同目录 `.uya` 属于同一模块，避免同模块重复定义。

## 3. TDD 工作流

每个功能默认按以下顺序推进：

1. 先写失败测试或可复现 smoke。
2. 再写最小实现，让目标测试变绿。
3. 最后重构，保持相关测试为绿。

每一步都要做自检：

1. 当前实现是否符合编译器现实能力，而不是概念设计。
2. 正常路径、边界路径、错误路径是否都有证据。
3. 改动是否引入了半成品接口、占位语义或未兑现的 TODO。
4. 如果发现设计不合理，立即回退并重写，而不是继续堆代码。

测试优先级：

- 纯单元测试：数据结构、编码、边界、错误处理。
- runtime smoke：可执行入口、协议解析、命令最小闭环。
- 集成测试：`redis-cli`/Python 驱动验证协议与持久化。
- 故障注入：RDB/AOF 损坏、截断、重启恢复、网络半包。
- benchmark/stress：吞吐、延迟、内存上界、恢复时间。

不接受“代码先写完，最后统一补测试”。TODO 子项只有在测试、实现、文档都同步后才可勾选。

## 4. Definition of Done

完成任何模块或 TODO 子项，至少满足：

- 有明确测试文件、测试组或验证脚本作为证据。
- 覆盖正常路径、边界路径和关键错误路径。
- `make build` 通过，相关 `make test` 或专项测试通过。
- 技术路线已被当前工具链验证，不依赖尚未证明可用的能力。
- 没有新增工程绝对路径。
- 文档、TODO 状态、命令示例与实际行为一致。
- 资源释放、回滚、内存统计或泄漏风险有明确处理。
- 性能相关改动给出 benchmark 口径；与 Redis 对比时必须注明硬件、命令、客户端并发、数据规模、持久化配置和统计口径。

后续应建立 `docs/redis-uya-definition-of-done.md`，把每个里程碑能力映射到具体测试或报告证据。

## 6. 版本策略

- redis-uya 项目版本统一写作 `vX.Y.Z`；预发布写作 `vX.Y.Z-alpha` / `vX.Y.Z-beta`，范围用完整版本号加文字说明表达。
- 当前内置编译器为 `./uya/bin/uya` v0.9.4 本地修复快照。
- Uya v0.9.5 未正式打 tag 前，不依赖 v0.9.5-only 特性。
- v0.9.5 发布后再升级 `uya/` 内置编译器与标准库，并补一次兼容性验证报告。

## 7. Redis 实现策略

- Phase 0 优先完成可编译骨架、测试框架、日志、时间、字节序、CRC64、分配器。
- 数据结构先行：SDS、Dict、Object、底层集合结构必须有单元测试和内存测试。
- 协议闭环优先：RESP parser、command router、PING/SET/GET 先打通，再扩展命令族。
- 持久化先可恢复：RDB/AOF 先保证兼容子集、校验、截断安全，再优化后台 rewrite。
- 复制和集群后置：先做稳定单节点，再做主从 PSYNC，基础集群功能最后收敛。
- 性能优化靠证据：Dict lookup、SDS append、RESP parse、网络事件循环必须用 benchmark 指导优化，最终以同条件 Redis 对照作为超越目标。

## 8. 验证命令

常用入口：

```bash
make version
make build
make run
make test
```

所有命令应保持相对路径可执行，不要求开发机存在固定外部工程路径。
